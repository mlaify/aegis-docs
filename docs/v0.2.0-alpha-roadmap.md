# Aegis v0.2.0-alpha Roadmap

## Overview

v0.2.0-alpha promotes the hybrid post-quantum cryptographic suite from placeholder to production. The release ships end-to-end encrypted, post-quantum-resistant messaging across the full protocol stack: core crypto, relay, gateway policy, CLI, and SDK.

Future releases will build on this foundation to deliver a complete, deployable product.

---

### Cryptographic Suite

`AMP-HYBRID-X25519-MLKEM768-ED25519-MLDSA65-V1`

- KEM: X25519 ECDH + ML-KEM-768 (FIPS 203) → HKDF-SHA256 → XChaCha20-Poly1305
- Signatures: Ed25519 + Dilithium3 / ML-DSA-65 (FIPS 204)
- Security: ≥ 128-bit classical, NIST Level 3 PQ; secure if either classical or PQ primitive holds

### Phase 2 — Prekey publish + atomic-claim endpoints (shipped)

| Area | What shipped |
|------|-------------|
| **aegis-identity** | `generate_prekey_bundle(identity, count, prefix)`, `sign_prekey_bundle()`, `verify_prekey_bundle_signature()`, `OneTimePrekeySecret`, `PrekeyBundlePrivateMaterial`; resolver helpers `publish_prekey_bundle()` and `claim_one_time_prekey()` |
| **aegis-api-types** | `PublishPrekeysResponse`, `ClaimedPrekeyResponse` |
| **aegis-relay** | `consumed_prekeys` table extended with `unknown_prekey` published-check; new `one_time_prekeys` table; `Store::store_one_time_prekeys` (idempotent INSERT OR IGNORE) and `Store::claim_one_time_prekey` (atomic UPDATE…WHERE claimed=0); `prekey_routes::publish_prekeys` (`POST /v1/identities/:id/prekeys`, signature-verified, `IdentityWrite` scoped) and `claim_prekey` (`GET /v1/identities/:id/prekey`); audit operations `publish_prekeys`, `claim_prekey`, `consume_prekey`(unknown). 12 new tests across storage and routes. |
| **aegit-cli** | `aegit id publish-prekeys --relay <url> [--count N=10]` — generates a signed batch of ML-KEM-768 one-time prekeys, persists private halves to `<id>.prekey-secrets.json` (append-merge across runs), POSTs the bundle. |

### Phase 3 — Send-side integration (shipped — end-to-end forward secrecy)

| Area | What shipped |
|------|-------------|
| **aegit-cli** | `aegit msg seal` claims a one-time prekey via `claim_one_time_prekey()` when `--relay <url>` is configured and recipient supports hybrid PQ; substitutes claimed Kyber768 in the hybrid combine; stamps `used_prekey_ids` before signing. Graceful fallback on `PrekeyPoolExhausted` with stderr warning. New `--no-prekey` escape hatch. `aegit msg open` looks up the matching `OneTimePrekeySecret` in `<id>.prekey-secrets.json`, swaps it into `HybridPqSuite::for_recipient`, and after AEAD-verified decrypt atomically (tmp + rename) splices the secret out of the local pool. 6 new unit tests covering load + consume contract. |
| **aegis-sdk (TypeScript)** | `PublishPrekeysResponse`, `ClaimedPrekeyResponse`, `OneTimePrekeySecret`, `PrekeyBundlePrivateMaterial` interfaces. |
| **aegis-spec** | Conformance v0.3 phase 3 rows flipped to ✅. |
| **aegis-docs** | architecture.md §5.4 rewritten to describe all four links of the forward-secrecy chain (generate / publish + claim / seal / push + open + consume). |

**Forward secrecy is now realized end-to-end.** Each one-time prekey participates in exactly one KEM combine; the relay's phase-1 atomic enforcement prevents replay; recipient consume-on-success deletes the local secret so even compromise of recipient state after-the-fact cannot decrypt past messages whose secrets have been consumed.

## Versioning Policy

Aegis follows `MAJOR.MINOR.PATCH[-PRERELEASE]`. During alpha, breaking wire-format changes are allowed between minor versions. The `suite_id` wire field provides backward compatibility at the envelope level regardless of library version.

See `versioning-policy.md` for details.
